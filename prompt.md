# 角色设定
你是一位资深的鸿蒙（HarmonyOS）应用开发专家，精通 ArkTS 语言、ArkUI 框架以及最新的 HarmonyOS 6.0.2 API。你擅长模块化架构设计、系统底层状态监听以及后台服务的开发。

# 任务目标
请帮我开发一款名为“好梦（Good Night）”的熬夜辅助工具App。该工具旨在通过正向激励和设备状态监测，帮助用户养成良好的作息习惯。

# 核心需求与业务逻辑
1. **点数对赌机制（核心周期）**：
    - 设定为30天为一个周期。
    - 用户在周期开始时，需投入30个“虚拟点数”。
    - 每天夜间（如 23:30 到次日 06:00）系统会进行熬夜判定。如果没有熬夜，次日自动返还1个虚拟点数。

2. **模块化与解耦设计**：
    - 系统必须采用高度模块化的设计，做到 UI 层、业务逻辑层、数据/资产层、底层设备检测层 完全分离。
    - **特别注意**：虚拟点数管理模块必须设计为独立的接口（Interface/Service），因为在未来的版本迭代中，这个模块极有可能会接入真实的支付系统，将“虚拟点数”替换为“真实微信/支付宝扣款及返现”。请确保底层逻辑的替换不需要修改UI和其他核心判定逻辑。

3. **熬夜行为检测机制（HarmonyOS 6.0.2 API）**：
    - 需要调用鸿蒙相关API实现夜间的状态监测。
    - **亮屏检测**：监听夜间设备屏幕的亮屏/息屏广播事件（Screen On/Off）。
    - **传感器检测**：调用加速度传感器（Accelerometer）API，判断夜间手机是否有频繁的物理移动。
    - **判定逻辑**：如果用户在规定的睡眠时间内，出现频繁亮屏或手机有明显移动，则判定为“熬夜”，次日不返还点数；反之则判定为达标。

请你按照以下步骤为我提供开发方案和代码：

1. **架构与目录设计**：提供基于鸿蒙当前推荐架构（如Stage模型）的工程目录树，并说明每个模块/文件夹的职责（重点展示点数模块和传感器模块的解耦）。
2. **点数与资产模块（Assets Service）**：提供基于接口（Interface）的点数扣除与返还逻辑代码，并演示如何为日后接入“真实现金流”留出扩展点。
3. **熬夜检测服务（Sleep Monitor Service）**：
    - 提供调用 HarmonyOS 6.0.2 屏幕状态监听 API 的核心代码。
    - 提供调用 加速度传感器 API 获取运动数据的核心代码。
    - 提供一个简单的综合判定算法函数。
4. **UI界面（ArkUI）**：提供一个简洁的首页 ArkTS 代码，展示“当前周期”、“剩余/已返还点数”、“昨晚是否达标”三个核心数据即可。
